
现在很多人使用 AI 处理文档的方式，本质上仍然很低效。我们把 PDF、文章、笔记或会议记录上传到 ChatGPT、NotebookLM 这类工具里，然后提出问题。AI 会检索文件，找出相关片段，再生成答案。

这种方式能用，但问题是：下一次你问一个类似问题时，AI 通常又要重新搜索、重新拼接、重新理解。之前的整理结果没有沉淀下来，知识之间的关系也没有累积。每一次提问都像从零开始。

Andrej Karpathy 提出的 “LLM Wiki” 正是为了解决这个问题。它的核心思想是：不要让 AI 每次都临时搜索原始资料，而是让 AI 先把资料读一遍，整理成一个持续增长、结构化、互相链接的 Markdown 知识库。之后再提问时，AI 面对的不是一堆散乱文件，而是已经整理好的知识网络。

## RAG 的问题：能检索，但不积累

大多数文档问答系统使用的是 RAG，也就是 Retrieval-Augmented Generation，检索增强生成。

它的流程通常是：

1. 用户上传文档。
2. 用户提出问题。
3. 系统从文档中检索相关片段。
4. LLM 根据这些片段生成答案。

对简单问题来说，这种模式足够。但如果问题需要跨越五六篇文档，把多个概念、人物、地点、时间线或观点联系起来，RAG 就会显得吃力。

因为它每次都要重新找材料、重新拼接上下文。它没有真正维护一个长期知识结构，也不会把上一次整理出来的概念关系沉淀下来。

如果你是研究者，连续几周读论文，用普通 RAG 问问题时，AI 仍然像第一次接触这些论文一样重新开始。这就是瓶颈。

## LLM Wiki 的核心：把资料变成可持续演化的知识库

LLM Wiki 的做法是反过来的。

当你新增一篇文章、一本书摘、一份 PDF 或一段会议记录时，AI 不只是把它保存起来等待以后搜索，而是会主动阅读、提取关键概念，并把它整合进已有 wiki。

它会做几件事：

1. 为新概念创建页面。
2. 更新已有页面。
3. 把相关概念互相链接。
4. 在索引页中加入新的知识入口。
5. 如果新资料和旧内容存在冲突，标记出来。
6. 记录本次更新改动。

这样，知识库会随着资料增加不断变厚。重要的是，连接关系已经提前建立，综合工作已经提前完成。之后再提问时，AI 不再面对原始资料堆，而是面对一个组织过的知识系统。

Karpathy 的类比大概是：Obsidian 像 IDE，LLM 像程序员，wiki 像代码库。你不需要手写每一页 wiki，而是让 AI 负责写作和组织；你只需要控制输入什么资料、提出什么问题、检查输出质量。

## 整个系统有三层

LLM Wiki 可以拆成三个简单层次。

第一层是 raw，也就是原始资料层。

这里放 PDF、网页文章、会议记录、文本文件、Markdown 等原始文件。它们是 source of truth，AI 可以读取，但不应该修改。这样可以保证原始材料始终存在，后续如果 wiki 内容有问题，还能回到原始资料核对。

第二层是 wiki，也就是 AI 生成和维护的知识库。

这里是一组 Markdown 文件，包括首页、索引页、概念页、实体页、主题总结页、对比页等。页面之间通过链接连接起来，形成一个可以在 Obsidian 中浏览和可视化的知识网络。

第三层是 schema，也就是规则文件。

如果用 Claude Code，这个文件通常叫 `CLAUDE.md`。它告诉 AI：这个知识库的目的是什么、目录结构是什么、怎样处理新增资料、页面应该怎样格式化、回答问题时应该先看哪里、如何引用来源、遇到不确定信息时如何处理。

## 搭建需要什么工具

视频中使用两个主要工具：

1. Obsidian  
   它是一个本地 Markdown 笔记工具，用来查看和管理 wiki。它有图谱视图，可以直观看到页面之间的连接。

2. Claude Code  
   它作为 AI coding agent，负责读取本地文件、生成 Markdown 页面、更新索引和维护 wiki。

视频也提到，不一定只能用 Claude Code。只要工具能读写本地文件，也可以用 Codex、Cursor 或其他类似 coding agent。Obsidian 也不是强制的，本质上这就是一个 Markdown 文件夹，用 VS Code 或普通文本编辑器也能打开。Obsidian 的优势是图谱视图更直观。

## 第一步：创建 Obsidian Vault

安装 Obsidian 后，新建一个 vault。vault 本质上就是一个文件夹，视频中命名为 `LLM wiki`，并放在 Documents 目录下。

建好后，在 vault 里创建三个文件夹：

1. `raw`  
   放原始资料。AI 只读，不修改。

2. `wiki`  
   放 AI 生成和维护的结构化页面。

3. `templates`  
   可选目录。如果你想手动创建统一格式的笔记，可以放模板。但在这个教程里，页面主要由 Claude 生成，所以这个目录不是必须使用。

最终结构大概是：

```text
LLM wiki/
  raw/
  wiki/
  templates/
  CLAUDE.md
```

## 第二步：创建 `CLAUDE.md` 规则文件

`CLAUDE.md` 是整个系统的关键。Claude Code 打开项目时会自动读取它，并按里面的规则工作。

视频提供了一个 starter template。最重要的是里面的 purpose，也就是这个 wiki 的用途说明。

示例里用的是“规划一次日本旅行”。如果你研究的是 renewable energy，就改成对应主题；如果你想整理读书笔记，就改成读书知识库；如果你想做课程资料库，就改成教学相关目的。

视频强调：刚开始最需要改的通常就是 purpose 这一行，其他规则可以先沿用模板，后续再随着 wiki 增长继续调整。

`CLAUDE.md` 主要包含几类规则：

1. 知识库目的  
   说明这个 wiki 是为了什么主题服务。

2. 文件夹结构  
   告诉 AI 原始资料在哪里、wiki 输出在哪里、哪些目录不能改。

3. ingest workflow  
   说明新增资料后，AI 应该如何处理：读取资料、提取概念、创建或更新页面、更新索引、记录变更。

4. 页面格式规则  
   例如每个页面顶部要有摘要；关键结论要标注来源；页面之间要链接相关概念。

5. 问答规则  
   当用户提问时，AI 应优先查询 wiki，而不是直接从原始文件重新开始；回答中要引用来源；不确定时要明确说明。

## 第三步：用 Web Clipper 收集网页资料

视频中还安装了 Obsidian Web Clipper 浏览器扩展。它可以把网页文章转成 Markdown，然后保存到 Obsidian 或下载到本地。

示例场景是规划日本旅行。作者先找了一篇关于东京旅行的博客文章，用 Web Clipper 保存为 Markdown，再拖入 `raw` 文件夹。

视频也说明，资料不一定必须是 Markdown。PDF、文本文件、Markdown 都可以放进 `raw`。Claude Code 可以读取这些文件并进行整理。

## 第四步：让 Claude Code ingest 第一份资料

资料放入 `raw` 后，需要在终端进入 vault 目录，然后打开 Claude Code。

接着给 Claude 一个明确指令：

```text
I just added a new source to the raw folder. Please read it and update the wiki.
```

Claude 会读取新资料，分析内容，然后计划创建哪些 wiki 页面。视频中它根据东京旅行文章生成了不同页面，例如东京不同街区、景点、旅行建议等。

确认计划没问题后，让 Claude 继续执行。大约几分钟后，Obsidian 的 `wiki` 文件夹中就会出现结构化页面。

打开这些页面可以看到：它们不是简单复制原文，而是按主题拆分、摘要、链接和组织。再打开 Obsidian 图谱视图，可以看到节点和连接开始出现。

这也是 LLM Wiki 的重点：一份资料进入系统后，不只是被保存，而是被消化成知识结构。

## 第五步：加入第二份资料，观察 wiki 如何更新

视频继续加入第二份资料：一篇日本美食指南。

再次把它保存到 `raw`，然后让 Claude ingest：

```text
I just added new sources to the raw folder. Please read it and update the wiki.
```

这一次，Claude 不只是创建新页面，还会更新已有页面。比如之前已经有东京街区页面，新的美食资料可能会补充每个街区的餐饮信息、推荐食物和相关链接。

这说明 wiki 开始具备累积能力。新资料不会孤立存在，而是会融入已有知识结构中。图谱视图里会出现更多节点和更多连接。

## 第六步：提出跨资料问题

视频中提出了一个需要跨资料整合的问题：

想住在既靠近好吃的地方，又靠近主要寺庙的街区，应该选哪里？

普通 RAG 可能会临时检索两篇文章，再拼答案。而 LLM Wiki 的回答来自已经生成的街区页、美食页、寺庙页等结构化内容。它可以把分布在不同来源里的信息连接起来，并引用具体 wiki 页面。

这就是它和基础 RAG 的明显区别：RAG 更像即时搜索；LLM Wiki 更像不断维护的研究笔记系统。

## 第七步：定期 lint wiki

视频还介绍了一个很关键的维护动作：lint wiki。

就像代码 lint 会检查代码问题，wiki lint 用来检查知识库结构和内容质量。

你可以直接要求 Claude：

```text
Please lint the wiki.
```

它会检查几类问题：

1. 页面之间是否有矛盾。
2. 有没有过时或可疑的说法。
3. 有没有孤立页面，也就是没有被其他页面链接到的 orphan pages。
4. 有没有断开的链接。
5. 有没有被提到但还没有独立页面的概念。
6. 引用是否完整。
7. 结构是否健康。

视频示例中，因为只导入了两篇文章，结构还比较简单，所以检查结果基本健康。但它仍然指出了潜在缺口，并提出可以修复引用问题。

这一步很重要，因为 AI 会犯错，也可能错误分类或建立不准确的连接。lint 不是装饰功能，而是长期维护 wiki 质量的必要手段。

## 适合哪些场景

视频给了几个典型使用场景。

对学生和研究者来说，可以边读论文、文章、资料，边让 AI 构建主题 wiki。最后得到的不是一堆高亮 PDF，而是结构化研究知识库。

对教师来说，可以把课程文件、教研材料、培训文档、教学文章放进去，逐步形成个人教学 wiki。

对企业来说，可以放会议记录、客户访谈、项目文档、新员工材料。新成员不用翻 Slack 历史或零散文档，可以直接浏览组织好的知识库。

对普通长期学习者来说，可以把书籍、播客、文章、学习笔记都放进去，形成个人百科。

只要是“知识会不断积累，并且希望它们被组织起来”的场景，都适合这个模式。

## 局限性

视频也强调，这不是魔法，有几个限制。

第一，它更适合个人规模。

Karpathy 提到的规模大概是 100 篇文章级别。如果你要管理几万页资料，仅靠 Markdown 文件和本地文件夹可能不够，需要更复杂的基础设施。

第二，输入质量决定输出质量。

如果你喂进去的资料本身低质量、过时或混乱，wiki 也会受到影响。你仍然需要筛选 source，而不是把所有东西无脑丢进去。

第三，需要一个能读写本地文件的 AI agent。

Obsidian 本身只是查看和管理 Markdown，不会自动理解资料、生成页面或维护连接。真正的引擎是 Claude Code、Codex、Cursor 这类 coding agent。

第四，AI 会出错。

它可能错误分类、错误连接概念，或者遗漏重要信息。所以需要人工检查，尤其是早期阶段。lint wiki 就是为了降低这类问题。

## 总结

LLM Wiki 的价值不在于“让 AI 回答一个问题”，而在于“让 AI 帮你持续维护一个会成长的知识系统”。

传统 RAG 每次都从原始资料中临时检索。LLM Wiki 则把资料提前整理成 Markdown 知识库，让概念、页面、来源和关系持续积累。它更像一个由 AI 维护的 Obsidian 知识库：原始资料保持只读，wiki 页面持续演化，schema 负责规定结构和行为。

对个人研究、教学、项目管理、读书学习和团队知识沉淀来说，这是一个很实用的 AI 工作流。它成本低、文件透明、数据保存在本地，而且能随着资料增加越来越有价值。
