# 面向 Agent 的知识库技术演进与范式迭代

在 AI Agent 的落地场景中，传统基于物理长度切片的检索增强生成（RAG）由于存在“语义断裂”与“多文档信息孤立”等缺陷，难以满足 Agent 进行复杂工具调用、反思和多步推理时对上下文精准度和完备性的极高要求。

知识库技术架构为此经历了从“一维物理切片”到“网状图谱组织”，再到“有状态知识编译”的范式演进。

## 一、 传统 Vector RAG 阶段：固定切片与一维检索

### 1. 设计思想

作为 RAG 的起点，其核心思想是**无状态的“粗筛与硬拼接”**。将长文档机械地按照固定字符或 Token 长度切分为物理块（Chunks），通过 Embedding 模型映射至高维几何空间，利用向量相似度进行近邻检索。

### 2. 核心机制

- **滑动窗口（Sliding Window）**：在固定大小切片的基础上，允许相邻文本块之间存在重叠（Overlap，通常为 10%~20%），用以缓解跨边界语义被切断的问题。
    
- **相似度计算**：利用余弦相似度等几何度量，对用户 Query 向量与库中 Chunk 向量进行匹配。
    
    $$\text{Similarity} = \frac{\mathbf{A} \cdot \mathbf{B}}{\|\mathbf{A}\| \|\mathbf{B}\|}$$
    

### 3. 局限性与 Agent 痛点

- **信息断裂与代词指代不明**：机械切片极易将完整的因果关系或上下文实体剥离，导致 Agent 召回的内容充斥着“它”、“该公司”等缺乏指代主语的噪声。
    
- **跨文档语义孤立**：Embedding 模型在编码时呈“单兵作战”状态。文档 A 与文档 B 之间错综复杂的引用、对比和演进关系无法在空间几何点中得到表达。
    
- **机械拼凑引发幻觉**：召回阶段仅将 Top-K 个孤立切片通过换行符生硬拼接，破坏了自然语言的逻辑网，Agent 面对相互冲突或具有时序演进的碎片时，极易产生逻辑混乱。
    

## 二、 高级语义优化策略：Late Chunking 与父子块解耦

为了在不改变向量检索架构的前提下最大化保留单篇文档内部的语义完备性，行业推出了前置增强与结构解耦方案。

### 1. 父子块策略（Parent-Child Documents）

- **机制**：解耦“检索单元”与“生成单元”。将文档切分为包含完整大局观的**父块**（1000~2000 Token），在父块内部细切出高密度的**子块**（100~200 Token）。向量库仅对子块做 Embedding。检索时若命中子块，则通过元数据（Metadata）映射关系，自动将对应的父块还原并输入给 Agent。
    
- **解决的痛点**：既保留了短文本向量特征集中、检索精准的优势，又避免了给 Agent 的上下文造成片段化缺失。
    

### 2. 基于 Embedding 的动态语义切片（Semantic Chunking）

- **机制**：完全抛弃固定长度切分，利用模型感知语义边界。先将文本拆分为原子单句，利用滑动窗口计算相邻句子的向量距离。
    
    $$\text{Distance} = 1 - \text{Cosine Similarity}(\mathbf{V}_i, \mathbf{V}_{i+1})$$
    
    当距离曲线出现统计学断层（如超过特定百分位数阈值）时划分边界，确保每个 Chunk 在主题上高度纯净。
    

### 3. Late Chunking（延迟切片）

- **机制**：先对整篇长文档（如 32k Token）进行全量 Transformer Encoder 编码，利用长文本 Attention 机制让每个 Token 均融合前后文信息。编码完成后再进行物理切片，每个切片的向量由其内部 Token 向量均值池化（Mean Pooling）合成。
    
- **解决的痛点**：即使切片内部缺失显式主语，其向量也已自带全局上下文特征。
    

### 4. 密集上下文注入（Contextual Retrieval）

- **机制**：在切片入库前，调用轻量级 LLM 结合整篇文档为当前切片生成 1~2 句话的“背景说明”（如：表明所属章节、具体指代对象），将其作为前缀永久拼接在切片头部再进行 Embedding。
    

## 三、 Graph RAG 阶段：图增强网状知识组织与推理

针对**多文档之间交叉知识孤立**的物理限制，Graph RAG 实现了从“无结构文本检索”到“结构化知识推理”的跨越。

### 1. 设计思想

将知识的组织形式从“一维物理点”升维至“网状拓扑图”。它在入库阶段将大模型的能力前置，将整库文档重构为实体（Nodes）与关系（Edges）的网状逻辑体。

### 2. 核心机制

1. **实体关系抽取**：通过 LLM 批量异步阅读全量文档，提取出所有核心实体和逻辑连线。
    
2. **社区发现与群组摘要**：运行图算法（如 Leiden / Louvain 算法）对拓扑图进行全局聚类，并在离线阶段利用 LLM 为每个知识社区（Community）生成宏观摘要。
    
3. **两阶段混合检索（Hybrid RAG）**：
    
    - **全局检索（Global Search）**：通过查询社区摘要，回答宏观归纳性问题。
        
    - **局部检索（Local Search）**：结合向量特征和图的 1~2 阶邻居路径，抓取跨文档的显式因果链。
        

### 3. 局限性与 Agent 痛点

- **写放大与天价建库成本**：在入库（Ingestion）阶段需要调用 LLM 进行极大规模的抽取和摘要生成，Token 消耗呈指数级上升，写入延迟极高。
    
- **错误固化（Error Compounding）**：若离线抽取阶段 LLM 对关系的理解产生幻觉，错误关系将被永久“烧录”进图谱中，导致后续检索持续受污染。
    

## 四、 颠覆性 LLM Wiki 方案：从无状态检索到有状态连续编译

作为对传统 RAG 范式的激进颠覆，以 Andrej Karpathy 为代表推崇的 **LLM Wiki（知识库编译）** 方案，彻底取消了向量数据库与物理切片的概念。

### 1. 设计思想

核心理念是“编译知识，而非搜索碎片”（Compile Your Knowledge, Don't Search It）。它将多篇原始文档视为“源码（Raw Sources）”，利用 LLM 将其一次性增量“编译”成一套高度结构化、互相双向链接（Backlinks）的 Markdown 知识网络（Wiki 页面），实现系统状态从 Stateful 到 Continuous Evolution 的转换。

### 2. 核心机制

- **三层架构（Sources $\rightarrow$ Wiki $\rightarrow$ Schema）**：由一个明确的 `Schema` 指导文件（如 `CLAUDE.md`）约束 LLM。当新文档（Sources）输入时，LLM 作为“知识记账员（Bookkeeper）”，通读新文并直接重写、合并、修正已有的 Wiki 页面（Wiki Artifacts）。
    
- **模型自导航检索（Index-Driven Navigation）**：放弃几何向量匹配。检索时，系统首先将由 Wiki 页面提炼出的极简轻量级 `Index` 目录喂给长上下文 Agent，由 Agent 进行语言学逻辑推理，自主生成导航路径去读取确切的 Wiki 页面或文本块。
    
- **全生命周期维护**：内置查重、陈旧数据降权（Decay）、定期 Linting（一致性校验）机制。
    

### 3. 局限性与 Agent 痛点

- **规模墙限制（Scale Wall）**：由于严重依赖将核心索引和 Wiki 页面载入大模型的上下文窗口，当文档体量达到海量异构级别（如数百万历史工单）时，会导致 Context Window 溢出，并引发天价的在线推理费用。
    
- **编译吞吐量低**：每增加一个数据源，都需要触发复杂的 LLM 链式重写，不适合高频实时写入的动态数据流。
    

## 五、 面向 Agent 的知识库技术特性横向对比

|**特性维度**|**传统 Vector RAG**|**高级语义 RAG (Late/Parent-Child)**|**Graph RAG 流派**|**编译型 LLM Wiki**|
|---|---|---|---|---|
|**底层数学/逻辑依托**|高维向量空间几何距离|深度 Attention 融合 + 结构解耦|图拓扑网络 + 社区聚类算法|语言学逻辑导航 + 结构化 Markdown|
|**跨文档交叉语义支持**|**极差**（完全孤立）|**差**（依然受限在单篇物理边界内）|**极强**（显式逻辑连线构建知识网）|**极强**（LLM离线去重、合并与双向链接）|
|**消除代词/边界噪声**|弱（极易语义断裂）|强（通过上下文注入与父块还原）|中（依赖三元组提取精度）|**极强**（在编译阶段已被 LLM 消除和重写）|
|**建库计算开销（写）**|极低（仅需 Embedding 编码）|较低（轻量 LLM 辅助打标或长文本编码）|极高（需海量 LLM Token 离线提炼）|高（增量写入时存在显著的“写放大”）|
|**检索确定性（读）**|弱（依赖 Top-K 运气匹配）|中（细节召回率大幅提升）|强（顺藤摸瓜式的因果链路追踪）|**极强**（100% 确定性的逻辑路由导航）|
|**最适配的 Agent 场景**|简单的 QA 问答、长尾不活跃数据|结构独立、规则明确的技术/法律手册|错综复杂的关系审计、多主体对比与推理|核心 SOP 固化、高价值领域专家知识库|

## 六、 生产环境落地的分层架构共识

在实际工程架构中，单一流派很难同时兼顾“海量规模”与“极致语义交叉”。目前针对 Agent 的终极落地标准往往采用**分层 Hybrid 架构**：

1. **核心骨干层（顶层大脑） $\rightarrow$ 采用 LLM Wiki**：将数量可控（通常 $< 100\text{k}$ Token）的核心业务 SOP、关键政策、公司全局实体定义，交由 LLM 编译为有状态的 Wiki，作为 Agent 决策的核心记忆层。
    
2. **关系图谱层（中层脉络） $\rightarrow$ 采用 Graph RAG**：将多项目、多财报、存在强因果关联的文档织成知识图谱，承载复杂的网状跨文档推理任务。
    
3. **海量存储层（底层血肉） $\rightarrow$ 采用高级 Vector RAG（Late Chunking + 父子块）**：用于承载数以亿计的长尾历史日志、日志、历史谈话记录。通过 Hybrid 路由分发，需要细节时再由 Agent 发起向量捞取，实现整体系统的高效与降本增效。
